三、可写入简历的技术决策（因果链）

这几条是"遇到什么问题 → 原方案为何不够 → 怎么设计 → 怎么实现 → 可验证结果"最完整、最扛追问的。

1. 支付回调的幂等与并发（最强，建议放第一条）
问题：微信回调不保证只送一次，且对账轮询也会独立触发结算——重复触发若各加一次钱就会重复入账。
设计：单事务内先用数据库里不可变的订单金额列复核回调金额（防 TOCTOU，不信任回调携带的金额），再用 UPDATE payment SET status='SUCCESS' WHERE status='PENDING' 的 CAS——只有唯一命中者（count===1）累加已付金额并推进订单状态，重复回调命中 0 行，直接当幂等空操作返回。回调与对账轮询复用同一条结算路径，所以双触发天然安全。退款走 SELECT ... FOR UPDATE 行锁串行化，在锁内重算累计退款额拒绝超额退款；"取消后又到账"显式保留为人工退款信号。
证据：payment.service.ts:297-387（settle）、391-608（refund 锁）、757-798（退款对账轮询）；e2e：payments-refund.e2e-spec.ts(641 行)、payments.e2e-spec.ts(631 行)。

2. 支付/短信/存储的可插拔 Provider 抽象
问题：作品集要能演示、生产要接真实通道，且不能上线时大改业务代码。
设计：每类外部依赖 = TypeScript 接口 + Symbol 注入令牌 + @Global 模块的 useFactory 按环境变量选实现；业务代码只注入接口令牌，切换 Provider 零业务改动。两层 fail-closed：生产启动校验拒绝 Mock（env.validation.ts）、Mock/本地实现构造函数在 NODE_ENV=production 直接抛错。真实实现是硬货不是桩：微信支付 V3 用 node:crypto 自己做 RSA-SHA256 签名/验签 + AES-256-GCM AEAD 解密。
证据：payment-provider.module.ts:18-27、sms.module.ts:12-21、storage.module.ts:18-36、env.validation.ts:194-266、wechat.payment-provider.ts(581 行)。

3. 订单状态机 + 归属安全 + 服务端定价
问题：状态乱迁移会造成脏数据；下单归属若靠手机号兜底会串号；金额若信客户端会被篡改。
设计：9 状态穷举转移表 + assertTransition，所有变更收敛到单一入口，事务内乐观锁 CAS 更新并原子写状态历史；归属只认已验证 JWT 的 customerId，匿名不绑账户，客户查询强制并入 customerId，越权/不存在统一 404 防枚举；下单金额一律服务端按套餐/活动重算。套餐/活动"二选一"用 DB CHECK 约束 + 应用层 + mapper 三层守。
证据：order-status.constants.ts:8-27、orders.service.ts:199-222、order-checkout.service.ts:75-136,109、迁移 chk_order_must_have_target。

4. 前后端共用 Zod 的端到端类型契约
问题：前后端各写一套 DTO/类型，容易漂移。
设计：@repo/shared 的 Zod schema 作单一事实源——后端 nestjs-zod 的 createZodDto + 全局校验管道复用它，前端表单与提交前二次校验复用它，再由 openapi-typescript 从 API 的 OpenAPI 文档生成类型安全客户端；CI 用 git diff --exit-code 卡生成物漂移。
证据：packages/shared/src/schemas/*、content/packages/dto/package.dto.ts:13、app.module.ts:104、ci.yml:73-91。

5.（可选）认证与防爆破
验证码按手机号+IP 双维度限流、失败锁定、IP 维度失败计数抵御换号爆破、常量时间比对；刷新令牌用 Redis 轮转家族 + 重放检测（发现盗用吊销整族）+ 10s 宽限期兼容多标签页。
⚠️ 风险点：JWT 是手写 HS256（node:crypto HMAC），没用成熟库。面试官很可能问"为什么不用 jsonwebtoken/passport"。要么准备好答案（学习目的/零依赖/但生产我会换成熟库并说明理由），要么这条降级只写限流与令牌轮转、不强调"手写 JWT"。
