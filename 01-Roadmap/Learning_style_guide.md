## 一、本文件的目的

原 Roadmap 的 Prompt 是按「**澳洲 Senior 工程师视角**」设计的，对 Graduate 起步的我来说**起点过高**——具体表现：

- 第一层目录直接跳 Senior 级 topic，看不到"基础语法"层
- 代码示例 20-60 行，初学者容易迷失
- 大量未翻译的英文术语（generic / narrowing / inference 等），需要边看边查
- 面试题密度太大，新手做不完

本文件**不改变 Roadmap 的学习目标**（80% 技术栈、Tier 1/2/3 清单、刷题策略全部保留），
只调整 **Claude 输出的「粒度、节奏、术语处理」**，让 Graduate 跟得上。

## 二、学习者画像

| 维度 | 当前状态 |
|---|---|
| 编程基础 | 会写 JavaScript，但 TypeScript 语法很多看不懂 |
| 英文术语 | 大部分需要中文解释（generic / union / narrowing 等） |

## 三、第一层输出调整（Outline）

### 调整规则

1. **拆分入门/进阶/高级三个区段**，每段 4-5 个 topic，**用视觉分隔（🟢🟡🔴）**
2. **入门区起点降到"看懂 JS 代码就能跟"**：从最基础的语法开始
3. **每个 topic 之间的难度跨度变小**——宁可多拆几个 topic，也不要一个 topic 涵盖太多
4. **每个 topic 加「先修」字段**：明确告诉学习者需要先懂哪几个前置 topic
5. **每个 topic 的「为什么 Senior 要会」改成 2 句话**：第一句讲日常用途，第二句讲面试/架构价值

### 示例（TypeScript 的新版第一层）

> 已在聊天里给出过，作为参考标准。要点：
> - 🟢 入门区（Topic 1-5）：基本类型 / type vs interface / 联合交叉 / 函数类型 / 对象类型
> - 🟡 进阶区（Topic 6-9）：类型守卫 / 泛型基础 / 枚举字面量 / 模块与 tsconfig
> - 🔴 高级区（Topic 10-13）：工具类型 / 条件类型+infer / 映射+模板字面量 / 严格模式

### 输出后的引导问题

不要只问"要展开哪个 topic？"，改成：

> 要展开哪个 topic 的第二层？
> - 输入序号（如 `2`）：展开单个 topic
> - 输入 `all`：一次输出所有 topic 的第二层（信息量大，建议先看完蓝图再选学）
> - 输入 `术语 XXX`：先做术语扫盲（适合连目录里的词都看不懂时）

## 四、第二层输出调整（Knowledge Points）

### 调整规则

1. **入门区 topic 的知识点控制在 5-7 个**（原来 5-10 个），每个更小更聚焦
2. **每个知识点加「⚠️ 前置知识」**：告诉学习者需要先懂什么（哪怕是本 topic 内的前置知识点）
3. **标题用口语化中文**：
   - ❌ 不要："类型兼容性"
   - ✅ 改成："为什么 `string` 不能赋值给 `number`"
4. **每个知识点加「估计时长」**：例如 ⏱️ 15 分钟 / 30 分钟 / 1 小时
5. **进阶/高级区 topic 维持原 5-10 个知识点**

### 输出后的引导问题

> 要深挖哪个知识点的第三层？
> - 输入序号（如 `2`）：展开学习卡片
> - 输入 `按顺序`：从知识点 1 开始依次展开（适合系统学习）
> - 输入 `先看代码`：跳过铺垫直接看实例代码（适合已经懂个大概的）

## 五、第三层输出调整（Learning Card）

### 整体结构（从原版 5 个 section → 改为 6 个 section）

新增 **Section 0「术语扫盲」**，放在最前面。

```
0. 📖 术语扫盲（⭐ 新增）
1. 📝 实例代码（含「代码读法说明」⭐ 新增）
2. 🎯 使用场景
3. 🔄 可替代的类似技术
4. 💼 面试常考题目（数量按难度递增 ⭐ 调整）
5. 🧪 小测验
```

### Section 0：📖 术语扫盲（新增）

**为什么加这个**：Graduate 阶段最大障碍是英文术语，每次现查打断学习节奏。

**输出格式**：

```markdown
### 0. 📖 术语扫盲

在读下面的代码前，先认识这些词：

| 英文术语 | 中文 | 一句话解释 |
|---|---|---|
| Generic | 泛型 | 写函数/类型时先不指定具体类型，用 `T` 占位，调用时再决定 |
| Inference | 类型推断 | TS 不用你写类型，它自己猜出来 |
| Narrowing | 类型收窄 | 一个变量原本可能是 `string \| number`，经过 if 判断后 TS 知道它就是 `string` |
```

**规则**：
- 只放本卡片代码里**实际出现**的术语，不要堆砌
- 每张卡片 3-6 个术语为宜
- 如果某个术语之前的卡片讲过，加注 "（Topic 3 学过）"，不重复解释

### Section 1：📝 实例代码（调整）

**变化点**：

1. **代码前先写「代码读法说明」**：用 2-3 句话告诉学习者这段代码每一段在干嘛
   ```markdown
   📌 代码读法说明：
   - 前 5 行定义了一个 User 类型（数据形状）
   - 中间的函数接收一个 User，返回它的全名
   - 最后两行是调用示例
   ```
2. **代码长度从 20-60 行 → 改为 15-30 行**（入门区）/ 20-40 行（进阶区）/ 30-50 行（高级区）
3. **代码里的中文注释更详细**，几乎每一行都标注作用（入门区）
4. **保留原版的「⚠️ 易错点」标注**

### Section 2-3：使用场景 + 替代技术（保持不变）

原版规则保留。

### Section 4：💼 面试常考题目（按难度调整数量）

| Topic 难度 | 面试题数量 |
|---|---|
| ⭐ 入门区 | 1 题（🟢 基础 难度） |
| ⭐⭐ 进阶区 | 2 题（🟢 + 🟡） |
| ⭐⭐⭐ 高级区 | 3 题（🟢 + 🟡 + 🔴） |
| ⭐⭐⭐⭐ Senior 必备 | 3 题（全部 🔴） |

**理由**：入门阶段塞 3 道面试题会让学习者焦虑。先打好基础，到进阶再加压。

### Section 5：🧪 小测验（保持不变）

原版规则保留。

### Section 末尾：🌟 Senior 思维彩蛋（保持不变）

**但增加一条**：如果该 topic 是入门区（⭐），彩蛋内容要**点到为止**，不要展开讲——
否则 Graduate 看不懂只会增加挫败感。例如：

- ❌ 入门区 不要写："Senior 在做 Repository Pattern 时会用泛型约束 + 条件类型组合，确保编译期类型安全……"
- ✅ 入门区 改成："Senior 不写 `any`——这是 mid 和 senior 在 PR review 时最常被指出的点。Topic 13 会专门讲怎么避免 any。"

## 六、退出机制（什么时候删掉本文件）

本文件是**临时辅助轮**。当下面任一条件满足，就可以考虑删除：

- [ ] 累计打卡 ≥ 20 个 topic（Tier 1 基本覆盖）
- [ ] 能脱稿读懂英文官方文档（React、NestJS 官网不用查词）
- [ ] 连续 5 张学习卡片都觉得「术语扫盲多余」
- [ ] 主动想看 30 行+ 代码示例，觉得短代码不够用了

**删除方式**：直接从 Project Knowledge 移除本文件，Claude 会自动回到原 Roadmap Prompt。

**渐进退出方式**（推荐）：
1. 第一阶段：删掉「术语扫盲」section（保留其他调整）
2. 第二阶段：代码长度放宽到 20-50 行
3. 第三阶段：恢复原版面试题数量（2-3 题/卡）
4. 第四阶段：整个文件删除
