# 🎤 RepoPilot 面试语料

> [!important] 用法
> 现成话术，练到能自然说出。配合 [[Agent 面试八股与答法]] 用。

## 90 秒项目自述（开场定基调）
> RepoPilot 是一个面向真实 GitHub 仓库的 issue resolution agent：给它一个仓库和一个 issue，它自动分析结构、定位代码、生成计划、改文件、跑测试验证、独立复查，最后输出 diff。**它全程手写、不用任何框架**——工具调用循环、权限闸门、结构化输出、状态机都是裸 SDK 实现的。核心设计是一个 **Plan→Execute→Verify→Review 的闭环**，其中"验证"是灵魂：改前跑测试存基线、改后再跑对比，用测试结果当法官而不是信模型说"我改好了"。目前深支持 Python，通过 adapter 保持框架无关。一个诚实的局限是我只做了小规模冒烟验证，下一步是在 SWE-bench Lite 上跑出成功率数字。

## 框架三句话（"为什么不用 LangChain"）
1. **懂**：LangChain 卖多模型/多工具/多向量库的胶水；LangGraph 卖状态图的持久化和可恢复。
2. **判断**：我这项目是 1 模型 + 9 工具 + 1 状态机，框架的抽象层在这规模上是净负担，还会把我最想理解的机制黑盒掉。
3. **不教条**：真需要长任务可中断续跑时，那不过是 50 行显式状态机；团队生产环境我会专门评估 LangGraph 的 checkpoint 层，但工具和 prompt 保持显式。
> 姿态：不贬低框架，"我理解权衡、我做了选择"。

## 金牌 bug 故事（"讲一个你调过的 bug"）
> 我的路径隔离函数 `jail()` 有段逻辑会剥掉"仓库名前缀"（继承自沙盒版本）。在 tinydb 上炸了——因为很多 Python 项目顶层包目录名就等于仓库名（`tinydb/tinydb/`），它把合法路径 `tinydb/queries.py` 剥成了不存在的 `queries.py`，导致每次 read/edit 都失败，agent 被逼着绕去用 sed 硬改。**而我的单元测试没抓到**——因为 pytest 的 tmp_path 是随机目录名，永远不会跟仓库名撞上，这个坑只有对真实仓库才暴露。修法是"先信字面路径、剥完不存在才剥"，并补了一条同名目录布局的回归测试。
> 亮点：定位到根因、解释了为什么测试漏掉、给了带回归测试的修复。

## 评测故事（补完 SWE-bench 后用）
> 我用 `run.jsonl` 执行轨迹做评测：成功率看每个 run 最后 report 事件的 ok 字段，平均工具调用数就是数 tool 事件。在 SWE-bench Lite 上跑了 N 条，修好 X 条，平均每条 Y 次工具调用、Z token。

## 可以甩出来的一句话原理
- **健壮的 agent 从不因一次失败而死，它把每次失败变成喂给下一步的信息**（[[错误即信息]]）
- **能用提示词哄的地方都是它可以不听的地方，真要拦必须用代码**（[[硬约束vs软约束]]）
- **给裁判看被告的自辩 = 裁判被带偏；只给证据才有真判决**（[[隔离即去偏见]]）
- **廉价假设 + 后续验证，胜过昂贵的前期确定**（[[廉价假设加后续验证]]）
- **黑名单是列举坏的（必输），沙箱是限制爆炸半径（能赢）**
- **kind 只能被 log，绝不能被 branch——这就是框架无关的判据**

## 诚实的局限（主动说，别被问出来）
- 只做了小规模冒烟，缺规模化评测数字（正在补 SWE-bench Lite）
- 检索用 ripgrep 词法、非向量（代码要精确符号匹配，这是有意选择）
- 安全靠黑名单+回滚，强隔离（Docker）还没做
- PR 集成是外壳雏形，还没打磨
