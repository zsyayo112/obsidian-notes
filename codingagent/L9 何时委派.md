# L9 · 何时委派（parent 怎么"知道"用 spawn_agent）

> [!abstract] 核心
> parent 没有任何"天生知识"。**它的行为 = 你给的工具 + system prompt，没有第三样。** 只给它 spawn_agent 并叫它别自己读，它才委派；给它直接读写工具，它就自己干了。

## 受控实验的结论

给 parent 同时配"直接 read_file"和"spawn_agent"，提示词保持中立，看它选哪条：

| 任务 | 直接读 | 派子 agent | 结果 |
|------|--------|-----------|------|
| 小（读一个文件） | 1 | **0** | 自己干 |
| 大（19 个文件） | 19 | **0** | 还是自己干 |

**即使大任务，中立提示词下它也宁愿自己硬读，绝不委派。** 模型走阻力最小的路。

## A/B 铁证（同一任务，两种做法）

| | 做法 | parent 上下文峰值 |
|--|------|------------------|
| [[L7 子agent与上下文隔离]] | 派子 agent | **2631 tokens** |
| L9 | 自己硬读 19 文件 | **17476 tokens** |

同样的信息、同样的答案，不隔离 → 上下文膨胀 **6.6×**。
子 agent 的价值不是"优雅"，是**可扩展**：让上下文开销不随任务规模爆炸。文件再多几倍，自己硬读就撞上限、触发压缩（[[L4 上下文管理]]）甚至失败。

## 两种设计风格

- **纯协调者**（L7）：parent 只有 spawn_agent，凡事都派。上下文永远干净，但小事也有子 agent 的固定开销。
- **混合型**（Claude Code，也是成品包的做法）：parent 自己直接读写，只在大调查时才派。省小事开销，只在值得时隔离。

## 落到那条主线

> [!important]
> 想改变模型行为，只有两招：**调工具（硬约束）** 或 **调提示词（软约束）**。
> 想让它大任务时真去委派：要么把直接工具拿掉（硬，L7），要么在 prompt 里【明确】写"多文件调查必须用 spawn_agent"（软）。中立提示词不够——L9 证明了。
> 见 [[硬约束vs软约束]]。

**不要为了用子 agent 而用子 agent**——这是学完编排最容易犯的毛病。单 agent 能干且不撑爆上下文时，直接干最省事。
