# 💡 廉价假设 + 后续验证

> [!important] 一句话
> **别在还不确定值不值得时就砸钱买昂贵的确定性。先花小钱猜一个假设，把"确定对不对"的贵活儿留给下游验证——总成本反而更低。**

## RepoPilot 里的体现
`planner` 出计划时**只看目录树、看不到任何代码内容**，所以计划**可以猜错**。为什么这样对：
- 让 Planner 读文件求稳 → 该读哪些它还不知道，可能读一堆，成本无上限、不可预测。
- 只看目录树 → 成本是个固定的小常数。
- **Planner 的任务不是"准"，是"廉价地给个起点"**。核实假设对不对，交给随后会 search/read/跑测试的 `executor`——它反正要读文件才能改。让两人都读文件 = 为贵活儿付两遍钱。

## 比喻
医生先凭症状快速判断"八成是 X"（便宜），再做**那一项**针对性化验确认，而不是上来把所有昂贵检查全做一遍。

## 亲历
tinydb 那次，Planner 一个字节代码没读，只看 issue + 文件清单就猜中根因。它猜得准是运气，但**设计不依赖它猜准**——这才是关键。

## 面试
"怎么控制 agent 的上下文成本 / 怎么设计规划阶段" —— Plan-Execute 分离，Plan 只吃固定小常数上下文，故意可错、后续验证。
