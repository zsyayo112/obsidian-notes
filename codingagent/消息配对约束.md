# 🔗 消息配对约束

> [!warning] 一句话
> **messages 不是一个可以随便切片的列表。它有内部配对约束：每条 `assistant(tool_calls)` 后面，必须紧跟着与每个 `tool_call.id` 一一对应的 `tool` 消息。破坏它，API 直接 400。**

## 硬规则

- 一条 assistant 消息里可以有多个并行 tool_calls（每个带唯一 `id`）。
- 每个 id 都必须有一条 `role="tool"` 的结果消息用 `tool_call_id` 配对。
- 不能存在"无主"的 tool 消息（找不到对应 tool_call）。

## 什么时候会被破坏

- agent 循环被 `max_turns` 截断，最后一条 assistant 发了 tool_calls 但结果还没回。
- 上下文压缩时机没算准，从中间删了几条，正好拆散一对。
- 手动改历史。

## 两个防御手法

**① 压缩要整段替换，不能删几条**（见 [[L4 上下文管理]] 的 `compact`）：

```python
# 对：把 system 之后的整段历史压成摘要，替换掉
# 错：删掉最老的 N 条 —— 可能正好删掉半对
```

**② 每次请求前跑一遍 `sanitize()` 守卫**，剔除破损配对：

```python
def sanitize(messages):
    # 剥掉没有结果的 tool_call；丢弃无主的 tool 消息
    ...
messages[:] = sanitize(messages)   # 破损的配对永远到不了 API
```

这类**不变量守卫**是 agent 从"demo 能跑"走向"长时间不崩"的关键。
