# L3 · 手脚与沙盒

> [!abstract] 核心
> 给 agent `write_file` 和 `run_bash`，它就从"能看"变成"能干"，形成**写代码 → 运行 → 看错误 → 修**的闭环。同时，权限一放开，安全边界就必须自己画。

## 那个闭环的灵魂是一句提示词

```
写完代码后，必须用 run_bash 实际运行它来验证，不要只是声称它能工作。
```

删掉这句，模型会写完就宣称"已完成、可正常运行"——然后代码里有语法错误。**大部分"agent 不好用"的根子在提示词没要求它闭环验证。** 见 [[幻觉 模型看不见就编]]。

## 四道边界

1. 路径不许逃出沙盒：`resolve()` 后 `is_relative_to(PLAYGROUND)`（必须 resolve 再比，否则 `playground/../../etc` 前缀能骗过检查）。
2. 危险命令黑名单（**很弱**，永远列不全，只拦最蠢的意外）。
3. shell 的 `cwd` 锁死在沙盒。
4. 30 秒超时。

> [!warning] 黑名单不是安全机制
> `rm -rf /` 拦了，`rm -rf ~` 呢？`python -c "shutil.rmtree('/')"` 呢？真正的防线是沙盒目录 + 用户确认（[[L5 权限与硬约束]]）。

## 亲历的坑

`cd playground` 报错——模型以为要先进目录，其实 cwd 已经在里面了。**这是提示词裂缝，不是模型的错**：它严格按我给的（错误的）世界观行事。这个坑后面还会出现三次，见 [[路径困惑 一个反复出现的坑]]。
