# 🎛️ 手写状态机

> [!important] 一句话
> **orchestrator 是个显式状态机：一个 `while` + 若干状态。它自己不碰模型，只按顺序调度各模块。LangGraph 卖的就是这张图的框架版，手写不过几十行。**

## 状态流转
```
BASELINE → PLAN → EXECUTE → VERIFY ──重试──→ EXECUTE
                              │
                              └──→ REVIEW → REPORT → DONE
```
每个状态 = 调用一个专职模块：BASELINE→verifier、PLAN→planner、EXECUTE→executor、VERIFY→verifier.compare、REVIEW→reviewer。见 [[RepoPilot 架构大串讲]]。

## 为什么显式状态机，而不是一个大 while 循环
一次任务是几个**性质完全不同的阶段**，每个阶段结束都要做决策（继续？重试？转下一步？）。拆成显式状态后，"从 EXECUTE 出来只有两条路"一眼可见，且**每次状态切换都写进 trace**，事后能精确回放。

## 一条硬规则：DONE 是唯一终止态
所有结局（成功/失败/跑偏中止）都必须先经过 REPORT 这道唯一出口，把"发生了什么/diff/要不要回滚"讲给用户。**绝不允许"哑巴终止"**（把 state 一设、直接 return、什么都不交代）。
> 这条是我自己踩坑修出来的：原来"改动超限"直接 `state="FAILED"` 跳出循环，跳过 REPORT，用户看到程序突然退出、仓库脏着没人管。改成 DONE 唯一终止后，一切结局都被 REPORT 兜住。

## orchestrator 自己不碰 LLM
整段代码没有 `get_client`/`json_call`——它"用到"模型是通过调 make_plan/run_executor/review 间接实现。所以它是**纯确定性控制流**，属于[[两大类分层 碰不碰LLM|不碰 LLM]]那类，可被 trace 完整回放。

## 手写 vs LangGraph
LangGraph 给"状态图 + checkpoint（可中断可恢复）"。我这规模手写几十行更透、可调试；真需要"长任务续跑"再加个存盘也就 50 行。详见 [[LangGraph 是什么]]。

## 面试
"你怎么把 agent 各环节组织成可控整体" —— 显式状态机，每状态调一个专职模块，控制流确定可测、可 trace 回放，不确定性关在被调模块里。
