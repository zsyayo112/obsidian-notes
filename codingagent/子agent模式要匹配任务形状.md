# 🧩 子 agent 模式要匹配任务的「形状」

> [!important] 一句话
> **子 agent 的模式必须匹配任务的输入/输出形状。** 输入大、输出小（调查、检索、总结）→ 只读子 agent，隔离赢；输出大（建造、生成文件）→ 需要能写、且有独立工作区的建造者子 agent，否则隔离反而变成负担。

## 两种形状

| 任务形状 | 输入 | 输出 | 该用的子 agent | 效果 |
|---------|------|------|--------------|------|
| 调查（读一堆文件 → 总结） | 大 | 小 | 只读调查员（[[L7 子agent与上下文隔离]]） | **隔离赢**：脏活留在子 agent |
| 建造（一个规格 → 一个文件） | 小 | 大 | 建造者（能写 + 独立工作区） | 用只读的就**隔离亏** |

## 亲历：把调查员用去建造，翻车

让 3 个【只读】子 agent 去"写前端页面 + 后端 + 测试"。它们没有写工具，只能把整份代码当文本返回：

- 3 个子 agent 返回 4073 + 3389 + 3441 ≈ **11000 字符的代码全灌回主上下文**
- 主 agent 再拿这些文本自己 `write_file` → **同一份代码在主上下文里出现两遍**
- 还多花了 3 次子 agent 往返和 token

这恰好是 [[L7 子agent与上下文隔离]] 的**反面**：L7 是"大进小出"（隔离赢），这次是"小进大出"（隔离亏）。**子 agent 不是越多越好，用错形状是负收益。**

> [!warning] 反模式
> 别用只读调查员子 agent 去做建造任务。它写不了文件，只会把产物当文本吐回来，把你本想省下的上下文塞得更满。

## 顺带暴露的两个老问题

- 测试子 agent 任务含糊（给还不存在的文件写测试），就乱读 8 个无关文件、甚至 `list_files("/")` → [[L7 子agent与上下文隔离]] 说的"任务含糊 → 子 agent 瞎逛"。
- `cd playground` 第 6 次出现，连成品包 prompt 禁止也没用 → 最终在 `run_bash` 里用硬约束剥掉，见 [[路径困惑 一个反复出现的坑]]。

## 正确做法：建造者子 agent + 独立工作区

要让"并行建造"划算，得解决"多个子 agent 同时写文件会互相踩"（[[L8 并行子agent]]）：
- 给每个建造者子 agent 一份**独立工作副本**（子目录 / git worktree）
- 各自在自己的沙盒里写、跑、验
- 主 agent 最后把产物**合并**回主目录

这是 Claude Code 的做法，也是多 agent 编排里最接近工业界、难度最高的一块。

## 带走

> 学完编排最容易犯的毛病是"什么都想派子 agent"。先问一句：**这个任务是大进小出，还是小进大出？** 前者才是子 agent 的主场。见 [[L9 何时委派]]。
