# L7 · 子 agent 与上下文隔离

> [!abstract] 核心
> **子 agent 不是新东西，就是"再跑一遍那个 while 循环"，只不过用全新的 messages 和 system prompt。** 主 agent 的一个【工具】就是"开一个子 agent"。真正的新概念只有一个：**上下文隔离**。

## 为什么要它

回想 [[L4 上下文管理]]：读一个文件全文就永久驻留、token 暴涨。若主 agent 要通读很多文件，上下文当场爆炸。
子 agent 的解法：派它去读，**翻文件的几千 token 全烧在子 agent 自己的上下文里**，它只返回一句结论给主 agent。主 agent 上下文始终干净。

> [!important] 这才是 Claude Code 用子 agent 的首要原因——不是并行，是把脏活的上下文开销挡在主线程之外。并行只是附带好处。

## 亲历的数字（隔离的铁证）

一次调查，两个子 agent：
- 读 7 个文件 → 内部峰值 **8901 tokens** → 只返回 ~2633 字符
- 读 9 个文件 → 内部峰值 **7881 tokens** → 只返回 ~1304 字符
- **主 agent 自身上下文全程只有 2631 tokens**

对比 [[L9 何时委派]]：同样任务让主 agent 自己硬读，它的上下文冲到 **17476 tokens**（6.6×）。这就是隔离的价值。

## 亲历的坑（第一版翻车）

最初偷懒复用了 playground 沙盒的工具，子 agent 站在沙盒里却被要求查项目根，什么都没找到还凭文件名瞎编：
- 工具世界 ≠ 任务世界 → [[路径困惑 一个反复出现的坑]] 第 5 次
- 只 list 不 read 就下结论 → [[幻觉 模型看不见就编]]

修法：给子 agent 指向【正确世界】的只读工具，并在提示词里加"必须 read_file 再下结论"的护栏。

## 子 agent 的代价（它是黑箱）

1. 主 agent 只看到结论，看不到过程 → 拿**可观测性**换干净上下文。
2. 它不能回头问你，任务描述有歧义就自己脑补 → 又是 [[幻觉 模型看不见就编]]。
3. 固定开销：每个子 agent 都要重发 system prompt + 工具定义。

> [!tip] 编排的难点不在"怎么开子 agent"（就 8 行循环），而在"怎么把任务和工具交代对"。
