# 🎯 Agent 面试问答·完整答案

> [!important] 用法
> 每道题一段"能念出口"的答案（20–40 秒）。练到脱稿。开场自述/框架三句话/bug 故事/评测故事在 [[RepoPilot 面试语料]]。速查映射在 [[Agent 面试八股与答法]]。

## 基础机制

**Q：Agent 的本质是什么？**
> Agent = 一个大模型 + 一个 while 循环 + 一组工具函数。模型自己什么都不执行，它只输出文本；所谓"调用工具"，是模型吐一段表达意图的结构化文本，**我的代码**去真执行，再把结果塞回对话、重新问模型。循环往复，直到它不再要工具、给出最终答案。我的 executor.py 就是这个循环的实证。

**Q：工具调用底层到底怎么发生的？**
> 一个工具有两半：实现半是个普通 Python 函数（我的代码，模型看不见），说明书半是一份 JSON schema（发给模型看，模型只见这半）。模型看着 schema 吐出"我要调 read_file、参数是 x"，我的代码在 execute 里照实现半真执行。所以"模型调用工具"其实是模型表达意图、我的代码执行。见 [[工具设计 实现与schema两半]]。

**Q：agent 主循环怎么写？**
> 问模型 → 它要工具就执行、把结果喂回 → 再问，直到它这一轮只回文字、不再要工具，那就是"干完了"的信号。外加一个 MAX_TURNS 兜底防无限打转，还有流式下要按 index 把工具调用的碎片重组。见 [[三层循环 阶段-步骤-重试]]、[[流式输出与碎片重组]]。

## 安全与健壮

**Q：怎么防幻觉 / 怎么确认它真修好了？**
> 不信模型说"我改好了"，用测试当法官。改前跑一遍测试存基线、改后再跑、比差值——"修好了"是测量出来的事实。而且判断"成功"这件事我故意交给确定性代码（verifier），不交给会胡说的模型，等于不让考生批自己的卷子。见 [[验证是灵魂]]。

**Q：怎么防止它改出回归？**
> verifier 的 compare 里，"有没有出现基线里没有的新失败"这个检查排在最前面、压倒一切。哪怕它修好了目标 bug，只要弄坏了任何原本通过的测试，一律判 regressed。回归比"没修好"还坏，所以回归优先。

**Q：工具执行出错怎么处理？**
> 错误即信息。工具永远返回字符串、永远不抛异常——出错就返回一段错误文字喂回模型，它下一圈自己读懂、纠正、重试。抛异常会当场炸掉循环、永久失去自愈机会。这个思想在我项目里出现三次：工具出错、日志写失败、模型 JSON 不合法。见 [[错误即信息]]。

**Q：怎么保证 agent 执行命令的安全？**
> 分层防御：路径 jail 锁死在仓库内、危险命令黑名单、改动文件数上限、git commit/push 封锁（最后一步归人），加上 git 可回滚兜底。但我会诚实说清天花板：黑名单是"列举坏的"，永远列不全、挡不住蓄意；真正的强隔离要靠 Docker 沙箱"限制爆炸半径"。见 [[安全 黑名单vs沙箱]]。

**Q：怎么防止 agent 失控烧钱/改爆仓库？**
> 用硬约束，不用提示词祈求。MAX_TURNS 限循环圈数、MAX_MODIFIED_FILES 限改动规模、命令超时——都是代码里模型绕不过的检查。提示词里写"请别改太多文件"是软约束，模型可以不听。见 [[硬约束vs软约束]]。

## 检索、成本、评测

**Q：怎么控制 agent 的上下文成本 / 怎么设计规划阶段？**
> Plan-Execute 分离，且让 Plan 廉价。我的 planner 只看目录树、不读任何文件内容，所以它的成本是个固定小常数、且计划故意可以猜错——"确定对不对"的贵活儿留给随后会 search/read/跑测试的 executor。廉价假设 + 后续验证，胜过昂贵的前期确定。见 [[廉价假设加后续验证]]。

**Q：代码检索怎么做 / 用不用向量？**
> 我用 ripgrep 做词法检索，因为代码要精确符号匹配——搜 def __le__ 我要的就是这个 token，不是"语义相似"的代码。向量检索适合自然语言、措辞不对齐的场景，比如按语义搜文档，那要 embedding + 向量库 + top-k，生产里常和词法混合。这是有意的选择。见 [[RAG 向量检索vs词法检索]]。

**Q：结构化输出怎么做？**
> 逼模型进 JSON 模式（response_format），解析它，解析失败就把错误喂回让它自己改、重试。零框架。另一条路是 function calling，定义一个 submit 工具。能讲到"解析失败喂回重试"就比只答 response_format 深一层。见 [[结构化输出 JSON模式与重试]]。

**Q：怎么评测一个 agent？**
> 对每次运行的执行轨迹（run.jsonl）做聚合：成功率看每个 run 最后 report 事件的 ok 字段，平均工具调用数就是数 tool 事件，成本看 token。基准我用 SWE-bench——真实 issue、agent 出补丁、隐藏测试判分，Lite 300 条。我的 verifier 基线对比就是它判分逻辑的迷你版。见 [[SWE-bench与agent评测]]。

## 架构、多 agent、加分项

**Q：怎么把 agent 各环节组织成一个可控整体？**
> 显式状态机：一个 while + 若干状态（基线→计划→执行→验证→复查→报告），每个状态调一个专职模块，控制流本身确定、可测、可被 trace 回放，把不确定的模型调用关在被调用的模块里。而且 DONE 是唯一终止态，一切结局都过 REPORT，不允许"哑巴终止"。见 [[手写状态机]]。

**Q：怎么用 LLM 当可靠的裁判（LLM-as-judge）？**
> 上下文隔离 = 去偏见。我的 reviewer 是一次全新的独立调用，只喂它四样客观证据（issue、计划、diff、测试对比），屏蔽 executor 那句"我改好了"的自述。给裁判看被告的自辩，裁判会被带偏、审查退化成盖章；只给证据才有真判决。见 [[隔离即去偏见]]。

**Q：多 agent 怎么用 / 什么时候委派？**
> 子 agent 就是套娃的循环，用来把大块调查的上下文开销挡在主线程外。我实测过：并行独立子任务 2.9× 加速；中立提示词下模型绝不主动委派，隔离与否上下文峰值差 6.6×。约束子 agent 最硬的办法是只给它只读工具——想让它别乱写，就别给它写工具。见 [[L9 何时委派]]、[[子agent模式要匹配任务形状]]。

**Q：MCP 是什么？能接吗？**
> MCP 是 Anthropic 的开放协议，标准化 LLM 应用怎么暴露工具/数据/提示词给客户端，像"AI 工具的 USB-C"。我的项目是薄壳设计、核心逻辑都在 solve() 里跟命令行解耦，所以能不动核心就把它包成一个 MCP server，让 Claude Desktop 或 IDE 直接调。见 [[MCP 是什么]]。
