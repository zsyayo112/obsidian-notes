# ⚖️ 验证是灵魂：修好了要测量

> [!important] 一句话
> **"修好了"必须是测量出来的事实，不是模型说的一句话。改前跑测试存基线、改后再跑、比差值——把"成功判定"从会胡说的模型手里，夺给确定性的代码。**

## 为什么是灵魂
playground 里模型说"改好了"你就信；真实仓库里"看起来对"和"测试变绿"是两回事。让 LLM 判自己成没成功 = 让考生批自己的卷子，永远满分。所以 `verifier` 故意待在[[两大类分层 碰不碰LLM|不碰 LLM]]那一类——它是不会被模型忽悠的法官。这是"防幻觉"最硬的做法。

## compare 的五个状态（回归优先）
```
new_failures 非空？        → regressed   ① 最先查，回归最坏
基线红、现在绿？           → fixed
基线绿、现在也绿？         → still_green
失败数变少？               → improved
否则                       → no_change
```
**先查 new_failures（本来好的被改坏的）压倒一切**：哪怕修好了目标 bug，只要引入新失败，一律判 regressed。这是 agent 能否被信任去改真实代码的分水岭。

## 主判据是 exit_code，不是数字
pytest 输出 `214 passed` 这种格式，但 cargo/go/npm 格式完全不同、数字解析会失败；而**退出码是所有测试工具的通用约定**，永远可靠。这让它能[[框架无关|框架无关]]地工作。

## 面试
"你怎么防幻觉 / 怎么确认真修好 / 怎么防回归" —— 基线对比 + 回归优先 + 退出码当主判据。这是项目最强的说服力。
