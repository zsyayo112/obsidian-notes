# B1 · 订单 9 状态机

> [!abstract] 核心(简历原句)
> 「将订单生命周期建模为 **9 状态状态机**,以**显式转移表**约束合法迁移,并把**状态更新与状态历史写入同一事务**;结合**条件更新**处理并发迁移,避免旧状态覆盖新状态。」
> 四个考点:①状态机 ②显式转移表 ③同一事务 ④条件更新(CAS)防并发。

---

## 1. 什么是状态机
订单一生只能处于某几个确定状态之一,且从一个状态只能走到**特定的下几个**,不能乱跳(类比红绿灯 红→绿→黄→红)。「有哪些状态 + 允许怎么跳」= 状态机。

**9 个状态**(`schema.prisma` `OrderStatus`):
PENDING_PAYMENT 待支付 · PAID 已全额支付 · DEPOSIT_PAID 定金已付 · BALANCE_PAID 尾款已付 · CANCELLED 已取消 · REFUNDING 退款中 · REFUNDED 已退款 · COMPLETED 已完成 · CHECKED_IN 已核销(预留)

---

## 2. 显式转移表
`apps/api/src/orders/order-status.constants.ts:8`
```ts
export const ORDER_TRANSITIONS: Record<OrderStatus, OrderStatus[]> = {
  PENDING_PAYMENT: ['PAID', 'DEPOSIT_PAID', 'CANCELLED'],
  DEPOSIT_PAID:    ['BALANCE_PAID', 'CANCELLED', 'REFUNDING'],
  BALANCE_PAID:    ['COMPLETED', 'REFUNDING'],
  PAID:            ['COMPLETED', 'REFUNDING'],
  REFUNDING:       ['REFUNDED', 'PAID', 'DEPOSIT_PAID', 'BALANCE_PAID'], // 退款失败回滚到原付费态
  CANCELLED: [], REFUNDED: [], COMPLETED: [], CHECKED_IN: [],           // 空数组 = 终点站
};
```
读法:**左=当前在哪,右方括号=允许去的下一站,`[]`=终点站再也出不去。**

**四个剧本**(对号入座):
- A 全款:`PENDING_PAYMENT → PAID → COMPLETED`
- B 定金:`PENDING_PAYMENT → DEPOSIT_PAID → BALANCE_PAID → COMPLETED`
- C 超时:`PENDING_PAYMENT → CANCELLED`
- D 退款:`PAID → REFUNDING → REFUNDED`(失败则 `REFUNDING → PAID` 回滚)

**裁判**(`:23`):
```ts
export function assertTransition(from, to) {
  if (!ORDER_TRANSITIONS[from].includes(to)) throw new OrderInvalidTransitionException(from, to); // → 409
}
```
> [!tip] 为什么用集中转移表(面试可讲)
> = **单一事实源**:所有合法流转**一屏可审**;加新状态只改一处;避免散落各处的 `if-else` **逐渐长歪、互相不一致**(A 处允许 B 处忘拦)成 bug 温床。

## 3. 状态机引擎 `transitionOrder`(唯一合法变更入口)
`apps/api/src/orders/orders.service.ts:199-222`
```ts
async transitionOrder(tx, id, expectedFrom, to, opts) {
  assertTransition(expectedFrom, to);                    // ① 先过裁判(非法 → 409)
  const { count } = await tx.order.updateMany({          // ② 条件更新(CAS)
    where: { id, status: expectedFrom, deletedAt: null },//    ⭐ 关键:必须还是旧状态才改
    data:  { status: to },
  });
  if (count === 0) return { applied: false };            // ③ 没改到 → 竞态,安静退出(不抛)
  await tx.orderStatusHistory.create({ ... });           // ④ 改成功 → 同一 tx 写历史
  return { applied: true };
}
```

**逐点**:
- **`tx` 参数**:函数不自开事务,用调用方传入的 `tx`。→ ②改状态 和 ④写历史 共用同一事务 = **原子**(要么全成要么全撤)。落实简历「同一事务」。
- **⭐ `WHERE status = expectedFrom`(CAS / 乐观锁)**:把「检查」和「修改」合并成一条不可打断的原子 SQL。等于说「**只有它现在还是我以为的旧状态,才改**」。
- **`count`** = 实际改了几行:仍是旧状态 → 命中 `count=1`;已被别人改走 → 匹配 0 行 `count=0`。
- **`count===0` 安静退出**:返回 `{applied:false}`,**不抛错、不写历史**。这是良性竞态(别人先干了),不是错误。见 [[良性竞态 vs 真错误]]。

### 反面教材(为什么不能先查再改)
```
❌ 1.先查:现在是 PENDING 吗?→ 是   2.再改:设成 PAID
   两步之间有时间缝,别人在缝里改了 → 你基于过时判断覆盖 → bug
✅ 把「判断」塞进 UPDATE 的 WHERE,检查+修改一步原子完成
```

## 4. 双回调竞态时序(把机制串起来)
订单当前 `PENDING_PAYMENT`,微信回调 + 主动查单几乎同时到,都想改 `PAID`:
```
路径甲(回调)                    路径乙(主动查单)
UPDATE...WHERE status=PENDING    UPDATE...WHERE status=PENDING
  → 命中,count=1 ✅               → 此时已是 PAID,不命中,count=0
  → 改 PAID + 写历史 + 入账        → applied:false,安静退出 🤫
```
数据库**行锁让两条 UPDATE 排队**,只有先到的 `count=1`。→ 订单**只被推进一次**,绝不重复入账。

---

## 5. 为什么主动查单也要改状态?(对账补偿)
微信异步回调**可能丢**(网络/服务器重启)。若只靠回调:回调丢 → 订单卡在待支付 → 15min 后被超时取消 → **用户付了钱订单却被取消**(线上事故)。
所以要**主动查单兜底**:定期问微信「这单付了没」,查到已付就**替丢失的回调把订单推进**。
→ 简历「**支付通知与主动查单复用同一结算入口**」:两条路径最后都调同一结算函数(内部调 `transitionOrder`),逻辑一份、天然一致,靠 CAS 保证只生效一次。这也**正是并发的来源**。

---

## 6. 静默退出体现在哪(调用方)
`orders.service.ts` 超时扫描(`:288` 附近):
```ts
const r = await this.prisma.$transaction((tx) =>
  this.transitionOrder(tx, id, 'PENDING_PAYMENT', 'CANCELLED', {...}));
if (r.applied) cancelled++;   // applied:false → if 不成立 → 默默跳过,一声不吭
```
「静默」= ① `transitionOrder` 不 throw,只返回 `{applied:false}`;② 调用方看标记,false 就跳过。

## 7. 异常继承链(非法跳转 → 409)
```
assertTransition 查表失败
  → throw new OrderInvalidTransitionException(from,to)  // 带错误码+409+{from,to}
  → 继承 BusinessException → HttpException
  → 全局 Filter(关卡②)接住
  → 返回 HTTP 409 + { errorCode: 'ORDER_INVALID_TRANSITION', ... }
```

## 🎤 面试金句(40 秒)
> 「订单所有状态变更收口到一个 `transitionOrder`。它先用转移表校验合法性,再用一条**带旧状态条件的 `updateMany`(CAS 乐观锁)**做原子迁移——`WHERE status = 期望旧状态`。并发下数据库行锁保证只有一路 `count=1` 真改到,另一路 `count=0` 视为良性竞态安静退出。状态更新和历史留痕共用调用方传入的**同一个 tx**,保证原子。这样即使支付通知和主动查单同时到,订单也只被推进一次。」
