## A. 项目全局

### Q1「简单介绍下这个项目」
**答**:个人独立全栈的营地在线预订平台「双溪·关外」,**已部署真实生产**。业务三块:套餐内容展示、在线下单支付、发车统筹。技术栈 NestJS + PostgreSQL/Prisma + Redis + 微信支付 V3,前端 Next.js。我**主导架构、数据模型和技术选型**,重点做了支付幂等结算、退款并发控制和订单状态机。
> 三个锚点:**个人全栈 / 已上线 / 主导架构**。junior 描述产品,senior 描述自己在产品里的技术决策。

**追问链**:
- ➤ 「你一个人做的?AI 参与了多少?」→ 诚实框定:规格驱动 + AI 辅助的单人全栈,**我拥有架构/数据模型/每个技术决策**,用 AI 加速实现,通过多轮代码审查 + 生产验收把关。(比假装逐行手写更扛追问)
- ➤ 「为什么选 NestJS 不用 Express?」→ 需要 DI、模块化、装饰器路由、和 Prisma/Zod 生态契合,团队协作与分层清晰;Express 太裸,大项目易散。
- ➤ 「已上线有多少用户/订单?」→ 诚实:上线前阶段,站点 noindex 未公开,已完成微信真金 ¥0.02 + OSS 端到端验收。**不编用户量**。
- ➤ 「这项目最难的技术点是什么?」→ 支付回调的幂等与并发正确性(见 Q6/B2)。

## B. 数据模型

### Q2「一张订单为什么可能对应多条 Payment?」
**答**:一张订单可有多笔付款。`FULL` 订单=1 条(全款);`DEPOSIT` 订单=2 条(定金+尾款,二者互斥不同时出现全款);发生退款再 +1 条 `REFUND` 行,通过自关联 `refundOfPaymentId` 指回被退的原付款行。

**追问链**:
- ➤ 「退款为什么新插一行,不直接改原付款行状态?」→ 保留**完整流水**(可能部分退、多次退),原付款行是**不可变的事实**,退款是独立事件。和 OrderStatusHistory 同一种 append-only 哲学。
- ➤ 「定金和尾款怎么保证不重复付/不错付?」→ 每笔 Payment 有 `transactionType`(DEPOSIT/BALANCE/FULL)+ 幂等键 `idempotencyKey @unique`;订单状态机约束 DEPOSIT_PAID 才能进 BALANCE_PAID。
- ➤ 「怎么算一张订单已付了多少?」→ 订单上有 `amountPaid` 列(结算成功时累加),`balanceDue = totalAmount - amountPaid`。

### Q3「金额为什么用 Int 存,不用 Decimal/Float?」
**答**:两层。① 不用 float:`0.1+0.2≠0.3`,钱不能错。② 用 Int 存**分**(¥100=`10000`)而非 Decimal:JS 的 `number` 本身是 float64,金额出库在 JS 里算也会飘;整数分全程整数运算,零精度问题且更快更简单。

**追问链**:
- ➤ 「JS number 是 float64,存分会不会溢出?」→ 不会,`Number.MAX_SAFE_INTEGER` ≈ 9×10¹⁵,营地订单金额远够。
- ➤ 「前端展示要 ÷100,会不会又引入浮点?」→ `/100` 只做**显示**,不参与计算,安全。
- ➤ 「跨币种/汇率呢?」→ 本项目单币种(人民币),不涉及;若涉及会引入 minor unit + 汇率快照。

### Q4「OrderStatusHistory 为什么设计成只 insert、不 update(append-only)?」
**答**:审计/可追溯——每次成功的状态变更记一行(谁改、从什么到什么、为什么),**不可篡改**。
> [!warning] 常见陷阱(别答错)
> **防并发靠的不是历史表**,而是订单行上「带旧状态条件的原子 UPDATE(CAS)」。历史表负责**记录**每次被接受的迁移。别把「防并发」的功劳安到历史表头上——这会露怯。

**追问链**:
- ➤ 「那到底靠什么防并发?」→ CAS 条件更新(见 Q6)。
- ➤ 「append-only 数据一直涨怎么办?」→ 可按订单归档/冷存;当前量级无需。
- ➤ 「谁改的怎么记?系统操作呢?」→ `changedBy`:后台 userId / 客户 customerId(customer_cancel)/ `null`=系统或支付回调。

## C. 订单状态机(核心区,深挖高发)

### Q5「状态流转规则为什么做成显式转移表,而不是到处写 if-else?」
**答**:集中转移表 = **单一事实源**,所有合法流转一屏可审、review 一眼能查、加新状态只改一处;散落的 if-else 会**逐渐长歪、互相不一致**(A 处允许 B 处忘拦),成 bug 温床。

**追问链**:
- ➤ 「加一个新状态要改哪些地方?」→ 主要改 `ORDER_TRANSITIONS` 表 + 相关调用入口;有防漂移单测保证枚举顺序一致。
- ➤ 「非法跳转发生时返回什么?」→ `assertTransition` 抛 `OrderInvalidTransitionException` → 全局 Filter → HTTP 409 + 稳定错误码 `ORDER_INVALID_TRANSITION`。
- ➤ 「终态怎么表示?能不能从终态出去?」→ 空数组 `[]`(CANCELLED/REFUNDED/COMPLETED),`assertTransition` 里 `[].includes(to)` 恒 false,任何出边都被拦。

### Q6「举例:什么情况会两个请求同时改一个订单?你具体怎么防?」⭐(最高频)
**答(顺着因果说)**:
1. **场景**:用户付款后,微信**异步回调**把订单从待支付改已支付;同时我有**主动查单**补偿任务,查到已付也做同样的事。两者几乎同时到 = 并发。
2. **防法**:改状态用一条 `updateMany`,`WHERE` 同时带 `id` 和 `status = 期望旧状态`。数据库行锁让两条 UPDATE 排队——**先到的**订单还是待支付,条件命中改成功 `count=1`;**晚到的**订单已是已支付,条件不命中匹配 0 行 `count=0`。
3. **结果**:`count=0` 当良性竞态直接返回、不抛错不记账。订单只被推进一次。改状态 + 写历史用**同一事务**保证原子。

**追问链(往死里钻)**:
- ➤ 「count 为什么会是 0?谁让它变 0 的?」→(考因果)因为 `WHERE status=旧状态` 不满足 → 匹配 0 行 → count=0。**不是**「先知道 0 才知道被改」,是「条件不满足 → 0 行 → count=0」。
- ➤ 「去掉 `status` 条件、只按 id 改会怎样?」→ 两个请求各改一次、各记一次账 → **重复入账 / 订单被推进两次**。
- ➤ 「数据库凭什么保证两条 UPDATE 排队?」→ 行级锁:同一行的写操作互斥,先到的先拿锁改完提交,晚到的再执行时条件已不满足。
- ➤ 「为什么用乐观锁 CAS 而不是 `SELECT FOR UPDATE` 悲观锁?」→ 结算是短事务、冲突概率低,CAS 无需持锁、吞吐好;退款涉及重算累计金额、需读-算-写串行,才用 FOR UPDATE(见 B3)。**两处按场景选不同锁**。
- ➤ 「count=0 你为什么不抛错?」→ 它不是错误,是正常并发结果(别人先干了)。抛错会把良性竞态误报成故障、污染日志监控。**用返回值表达良性,用异常表达真错误**。
- ➤ 「如果两条 UPDATE 真的完全同时到呢?」→ 数据库串行化到行锁,不存在「真同时改到同一行」;二者依然一个 count=1 一个 count=0。
- ➤ 「幂等键在这里起什么作用?」→ 状态机 CAS 防「订单被推进两次」;Payment 层 `idempotencyKey @unique` 防「建出两条付款行」(P2002 冲突),两层配合。

### Q7「『同一事务』保证了什么?不在同一事务会怎样?」
**答**:改状态 + 写历史捆在一个 `tx`,要么全成要么全撤。若不在同一事务:可能「状态改了但历史没写」(程序在中间崩)→ 状态变了却无记录,对不上账。

**追问链**:
- ➤ 「事务是数据库层面还是应用层面?」→ 数据库事务,经 Prisma `$transaction`;`transitionOrder` 接收 `tx` 不自开,由调用方决定事务边界。
- ➤ 「事务里还做了别的吗(如入账)?」→ 结算路径里,推进状态 + 累加 amountPaid + 写历史在同一事务(见 B2)。
- ➤ 「事务隔离级别是什么?」→ Postgres 默认 Read Committed;CAS 的正确性靠行锁 + WHERE 条件,不依赖更高隔离级别。

### Q8「主动查单为什么也要改状态?会不会和回调打架?」
**答**:回调可能丢(网络/重启),只靠回调则丢了就「用户付钱订单被超时取消」。主动查单是**兜底补偿**,查到已付就替丢失回调推进订单。它和回调**复用同一结算入口**,靠 CAS 保证同时到也只生效一次——不打架,反而是并发的来源与防护对象。

**追问链**:
- ➤ 「查单多久一次?会不会太频繁?」→ 有 poller 间隔(env `PAYMENT_RECONCILE_INTERVAL_MS`,默认 120s),且有首查宽限/扫描窗口。
- ➤ 「首查宽限和超时取消有什么时序约束?」→ 铁律:补偿首查宽限必须 **< 待支付超时**,否则丢失回调会在订单被超时取消「之后」才补回,重开 paid-after-cancel 窗口。这条在 `env.validation.ts` 启动时强校验。
- ➤ 「取消后钱又到账怎么办?」→ 显式保留为**人工退款信号**(paid-after-cancel 由 Payment 层退款处理,不经状态机)。

## D. 架构与工程(地基)

### Q9「你的 API 怎么统一做参数校验 / 错误处理 / 响应格式?」
**答**:全局三件套(`app.module.ts:103`):`APP_PIPE` = ZodValidationPipe(进门验参,不合格 400);`APP_FILTER` = HttpExceptionFilter(业务 throw 的异常翻译成规范 HTTP 错误);`APP_INTERCEPTOR` = TransformInterceptor(成功响应统一裹信封)。业务代码只管做菜 + throw,脏活全局接管。

**追问链**:
- ➤ 「为什么用全局不用每接口写?」→ DRY、格式统一、改一处即可;每接口各写会不一致、易漏。
- ➤ 「异常怎么就变成 409 了?」→ 具名异常继承 `HttpException`,携带 HttpStatus + 错误码,Filter 认识它(见 Q10)。
- ➤ 「校验为什么用 Zod 不用 class-validator?」→ Zod schema 能**前后端共用做单一事实源**(见 B5),一份 schema 同时驱动运行时校验和编译期类型。

### Q10「自定义业务异常怎么设计的?为什么不用裸 HttpException?」
**答**:每类业务错误定义具名异常类,把「错误码 + 提示语 + HTTP 状态码」收口在一处;业务只写 `throw new XxxException()`。每个错误带**稳定机器可读错误码**(如 `ORDER_INVALID_TRANSITION`),前端据此精确处理,不用 match 中文。继承链 `BusinessException → HttpException` 让全局 Filter 天生识别。

**追问链**:
- ➤ 「前端怎么用这个错误码?」→ 按 errorCode 分支处理(如冲突提示、跳转),与文案解耦。
- ➤ 「Prisma 的 P2002/P2003 怎么进这套体系?」→ 全局 Filter 默认映射(P2002→409 冲突、P2003→400),特定场景在 service 主动 catch 转具名异常(如发车点被引用 P2003 → `PickupPointInUseException` 409 + 引导「改为停用」)。

### Q11「什么是依赖注入?你项目里怎么用?」
**答**:「谁来 new 对象」交给框架。框架启动时造实例(默认**单例**),谁在构造函数声明需要就把同一实例递进去。项目里满屏 `constructor(private readonly xxx: XxxService)`。好处 = **解耦 + 可测试**(测试塞 mock)。

**追问链**:
- ➤ 「为什么好测试?」→ 不自己 new 依赖,测试时能注入假实现,单独测这一层。
- ➤ 「单例会不会有并发/状态问题?」→ Service 无状态(不存请求态),并发安全;有状态的东西放 DB/Redis。
- ➤ 「Provider 怎么按环境选不同实现?」→ Symbol 注入令牌 + `useFactory` 按 env 选实现,业务只依赖接口令牌(见 B4)。

### Q12「Controller 里方法只有一行、全甩给 Service,为什么这么分?」
**答**:职责单一。Controller 只管 HTTP(接请求、转交),Service 管业务。→ 好测试(测业务不用起 HTTP)、好复用(网页下单/后台代下单共用同一 Service 方法)。

**追问链**:
- ➤ 「同一个 Service 真的被多个入口复用?」→ 是,如 PickupPointsService 同时被公开控制器和后台控制器用;下单结算被回调和主动查单复用。
- ➤ 「业务逻辑放 Controller 会怎样?」→ 无法复用、难测试、HTTP 与业务耦合。

### Q13「怎么防止内部字段泄露给前端?」
**答**:在 Prisma 查询用 `select` 白名单**从源头只捞需要的字段**(如公开发车点只取 id/name/address/sortOrder,不取内部 `notes`),而非捞回全行再删——数据压根没离开数据库。

**追问链**:
- ➤ 「除了 select 还有别的防线吗?」→ 出参 mapper 白名单二次收窄(双保险);越权查询统一 404 防枚举。
- ➤ 「敏感数据(手机号)呢?」→ 当前明文(作品集阶段,schema 已注释生产应 AES 加密);属诚实边界。
