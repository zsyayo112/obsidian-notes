# B0 · 项目地基(启动 · 配置 · 分层)

> [!abstract] 核心
> 一个 NestJS 后端 = **盖店(create) → 布置店(configureApp) → 开门(listen)**。请求进来后在内部走**三层**(Controller→Service→Prisma),进出各套**全局三关卡**(验参 Pipe / 兜错 Filter / 包装 Interceptor)。业务代码只管做菜和 `throw`,脏活全被全局机制接管。

---

## 1. 程序入口:`main.ts` 的三步骨架

`apps/api/src/main.ts`

```ts
const app = await NestFactory.create(AppModule, { rawBody: true }); // ① 盖店:按图纸 AppModule 造出餐厅
configureApp(app, config);                                         // ② 布置店
await app.listen(API_PORT);                                        // ③ 开门:开始监听端口接客
```

- `app` = 盖好的餐厅;`AppModule` = 图纸(列了 20 个模块)。
- `app.listen()` 不执行 → 店没开门,前端访问不到后端。

### 🔑 `rawBody: true`(全场最值钱的一行,`main.ts:19-22`)
让框架把**原始请求字节**存进 `req.rawBody`,给 `/api/payments/notify` **微信验签**用。
- **为什么必须要原始字节**:微信回调签名是对原始 body 算的。框架若先 parse JSON 再 stringify,字段顺序/空格一变签名就对不上。**验签必须拿未经加工的原始 body**。
- → 连到简历「微信支付 V3」:防伪造回调第一步就是验签,验签前提就是这个 rawBody。

### Swagger 生产硬门(`main.ts:28-34`)
```ts
const swaggerEnabled = SWAGGER_ENABLED && NODE_ENV !== 'production';
```
`/docs` 暴露所有接口形状,生产一律不挂;`NODE_ENV !== 'production'` 这道是**环境变量绕不过的硬门**。

## 2. `configureApp` 布置餐厅的 4 件事

`apps/api/src/bootstrap/configure-app.ts`

| 布置 | 代码 | 大白话 |
|---|---|---|
| 统一门牌前缀 | `setGlobalPrefix('api')` | 所有控制器路径前加 `/api` → `/api/orders` |
| 准入白名单 CORS | `enableCors({ origin: origins })` | 只放行白名单里的前端(`APP_WEB_ORIGINS`),防坏人网站盗刷接口 |
| 安全须知 helmet | `app.use(helmet(...))` | 一键贴齐浏览器安全响应头(HSTS/nosniff/frame-deny) |
| 信任门童 IP | `trust proxy = TRUST_PROXY_HOPS` | Nginx 反代后按真实层数取 `req.ip`,用于限流防刷 |

> [!warning] trust proxy 有安全深度
> 信任跳数配错的两个坑:值偏小(0跳直连却仍信 XFF)→ 攻击者伪造 XFF 旋转 IP **绕过限流**;值偏大 → `req.ip` 塌缩成反代地址 → **过度封禁**。所以做成受校验的 env,不硬编码。

## 3. 配置从哪来:环境变量 + fail-fast

`apps/api/src/config/env.validation.ts`

- **环境变量**:换环境就得变的值(如前端域名),**不写死在代码**,由运行环境启动时注入。代码里只有「登记 + 默认值」:
  ```ts
  APP_WEB_ORIGINS: z.string().default('http://localhost:3000,http://127.0.0.1:3000'), // :24
  ```
  真正的值在 `.env`(本地)或部署配置(生产),**不在代码仓库里**。
- **这个文件 = 开门前的验票员**:用 Zod 把所有配置挨个校验,少必需的/格式不对 → `throw` → 应用起不来(**fail-fast**,宁可开不了门也不带错配置偷偷营业)。
- **生产 fail-closed**(`env.validation.ts:191+`,B4 会细讲):`PAYMENT_PROVIDER must be "wechat" in production`——**生产禁用 Mock 收钱**。这是简历④「生产拒绝加载 Mock」的一半。

## 4. 分层架构:一个请求的三层流转

> [!note] 三个角色,职责严格分开
> ```
> 顾客 ──> 服务员 Controller ──> 厨师 Service ──> 仓库 Prisma/DB
>          接请求、看地址        业务逻辑、算钱      只管存取数据
> ```

| 角色 | 代码 | 负责 | 不负责 |
|---|---|---|---|
| 🧑‍💼 Controller | `*.controller.ts` | 接 HTTP、转交 | 不碰业务/数据库 |
| 👨‍🍳 Service | `*.service.ts` | 业务逻辑 | 不管 HTTP 长相 |
| 🧊 Prisma | `this.prisma.*` | 读写数据库(自动转 SQL) | 不懂业务 |

**样板**:`content/pickup-points/public-pickup-points.controller.ts`
```ts
@Controller('pickup-points')  // 门牌 → /api/pickup-points
@Public()                     // 免登录
export class PublicPickupPointsController {
  constructor(private readonly service: PickupPointsService) {} // ← 依赖注入
  @Get()
  list() { return this.service.findPublicActive(); } // 服务员只有一行:甩给厨师
}
```

- **Controller 薄如纸是好设计**:不算钱、不碰库,只转交。→ 好测试、好复用(网页下单/后台代下单共用同一厨师)。
- **依赖注入(DI)**:服务员没自己 `new` 厨师,只在构造函数「声明需要」,框架启动时**造一个实例(单例)递进来**。好处 = **解耦 + 可测试**(测试时塞假厨师 mock)。见 [[依赖注入 DI]]。

**厨师防泄露细节**(`pickup-points.service.ts:158`):
```ts
return this.prisma.pickupPoint.findMany({
  where: { isActive: true },
  select: { id: true, name: true, address: true, sortOrder: true }, // 🔑 故意不要 notes
});
```
`select` 让内部字段 `notes` **从数据库那步就不被捞出**,而非捞回再删——从源头防泄露。

## 5. 全局三关卡(每个请求自动经过)

`apps/api/src/app.module.ts:103-107`
```ts
{ provide: APP_PIPE, useClass: ZodValidationPipe },        // ① 进门验参
{ provide: APP_FILTER, useClass: HttpExceptionFilter },    // ② 出错兜底
{ provide: APP_INTERCEPTOR, useClass: TransformInterceptor }, // ③ 出门包装
```

| 关卡 | 时机 | 干嘛 |
|---|---|---|
| ① Pipe 验参 | 进业务层**之前** | 参数格式不对(如 headcount 传 "abc")直接拦,返 400。保证进厨师的数据干净 |
| ② Filter 兜错 | 业务 `throw` 时 | 把异常翻译成规范 HTTP 错误 + 统一 JSON。业务只管 throw |
| ③ Interceptor 包装 | 返回**之前** | 成功响应统一裹信封(如 `{ data: [...] }`) |

`APP_` 前缀 = **全局**,每个请求自动过,不用每接口单独写 → **DRY**、格式统一、改一处即可。

## 6. 具名异常收口(业务 throw → 变成 409)

`apps/api/src/common/exceptions/business.exception.ts`
```ts
export class OrderInvalidTransitionException extends BusinessException { // :80
  constructor(from, to) {
    super('ORDER_INVALID_TRANSITION',              // ① 稳定的机器可读错误码
          `Illegal status transition: ${from} -> ${to}`, // ② 给人看的信息
          HttpStatus.CONFLICT,                     // ③ HTTP 409
          { from, to });                           // ④ 附带数据
  }
}
```
- 继承链:`OrderInvalidTransitionException` → `BusinessException` → NestJS 内置 `HttpException`。→ 所以全局 Filter「天生认识它」,知道拿 409 当状态码。
- **设计价值**:业务代码只写 `throw new OrderInvalidTransitionException(from, to)`,干净;每个错误带**稳定错误码**,前端据此精确处理(不用 match 中文提示语)。关键词:**具名异常收口 + 稳定错误码**。
