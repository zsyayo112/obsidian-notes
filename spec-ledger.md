三、简历项目介绍

spec-ledger 是为 Claude Code 设计的 Agent Skill 插件，用于约束 AI 编程助手在已有代码库中按工单变更代码：先盘点已有实现避免重复造轮子，开放性技术决策逐项交给人签字确认，完成状态必须靠验收命令真实跑通才能标记，并配套零依赖 Node 脚本对 spec 生命周期状态做审计，接入 CI 自查。

四、简历核心描述

- 针对验收命令可能来自被第三方编辑过的文档、存在越权执行风险的问题，设计分级确认策略：仓库内命令直接执行，涉及网络/部署/删除等操作强制逐条人工确认，无法对应到具体验收项的命令直接拒绝执行。
- 定位到审计脚本的 frontmatter 解析器在 CRLF 换行文件上整行匹配失败、导致全部字段丢失的 bug，改为统一换行符后解析；为避免 git 归一化掩盖问题，测试改在运行时临时目录动态构造 CRLF 内容，并把 Windows 纳入 CI 矩阵。
- 发现 spec 生命周期在 skill 流程说明与审计脚本校验规则之间存在矛盾状态，设计新增 abandoned 终态并强制 superseded 必须绑定替代项引用，用测试覆盖两种终态的合法与非法组合。
- 为审计脚本和仓库自校验脚本（manifest 一致性、skill 描述长度、文档引用文件存在性）选择零依赖实现，只用 Node 内置模块手写 frontmatter 解析与测试断言，使脚本可直接嵌入任意项目无需安装步骤。
- 针对 AI 助手在已有代码库中容易重复实现或擅自决策的问题，设计 Skill 执行流程：写 spec 前必须引用亲自打开过的 file:line 证据，开放性技术选择整理成决策表由人逐项签字，未签字状态阻止进入实现阶段。

五、技术栈

- 核心语言与框架：Node.js（原生 ES Modules，无框架依赖）；Markdown + 手写 YAML 风格 frontmatter 作为数据格式
- 数据与基础设施：无数据库；用 git 历史作为验收阶段"计划 vs 实际"偏差记录的证据来源；GitHub Actions（Linux/Windows 矩阵 CI）
- AI 或 Agent 相关：基于 Claude Code Agent Skills 平台设计 Skill 流程文本（SKILL.md）与 AskUserQuestion 交互式确认机制；为 Agent 可执行命令设计信任边界策略（流程级，非运行时沙箱）
- 工程化与测试：零依赖冒烟测试（手写 check() 断言，无测试框架）、CI 矩阵测试、插件 manifest/skill frontmatter 一致性校验脚本、语义化版本 + CHANGELOG 维护

Bullet 2（CRLF bug）
1. 追问：根因具体是什么？为什么正则会失效？为什么不能直接提交一个 CRLF 测试文件？
2. 需掌握（我已经从 diff 反推清楚，你需要能复述）：JavaScript 正则里 . 默认不匹配 \r（属于行终止符）。原代码逐行 split('\n') 后每行仍带尾部 \r，(.*)$ 因为 . 吃不到 \r，永远到不了字符串真正结尾，导致整条 line.match(...) 失败——每一行都如此，所以"全部 key 丢失"。修复没有改正则，而是在读文件时统一 .replace(/\r\n?/g,'\n')。不能用提交的 CRLF 文件测试，是因为 git 默认会做换行符归一化，提交后 CRLF 会被转回 LF，测试就测不到这个 bug——所以测试改成在运行时用 .replace(/\n/g,'\r\n') 动态构造。
3. 代码位置：skills/spec/scripts/spec-status.mjs:63（正则未变）、:92（归一化）；skills/spec/scripts/test/run.mjs:5-8, 60。
4. 是否夸大：没有，这条最扎实，可以放心讲。
5. 无需替代表述，但建议面试时主动讲出"为什么 . 匹配不到 \r"这个细节，比只说"修了个换行符 bug"更有说服力。

七、面试讲解版本（1～2分钟口语稿）

"这个项目的起因是我在自己另一个项目里手动实践一套开发流程——每次接一个工单，先让 AI 把代码库里已经有的东西盘一遍，再决定要不要做、怎么做，做完之后必须真的跑一遍验收命令才算完成。我发现这套流程本身有价值，就把它从那个项目里抽出来，做成一个独立的、任何项目都能装的 Claude Code Agent Skill 插件，这就是 spec-ledger。

它的工作方式是这样：有两个 Skill，/spec 和 /spec-close。/spec 强制 AI 在写方案之前先搜代码库，把已经存在的能力用 file:line 证据列出来，避免重复造轮子；碰到有取舍的技术决策，会整理成一张表，一项一项让人确认，不允许 AI 替你拍板。/spec-close 负责收尾，它会真的把 spec 里写的验收命令跑一遍，失败了就把失败记录写进文档里，不会含糊过去；成功了才会记录"计划和实际做出来的东西之间有什么差异，为什么"。整个 spec 文档有一套状态字段，我配套写了一个零依赖的 Node 脚本去审计这些状态有没有问题，比如状态卡住太久没更新、标了完成但没有收尾记录之类的。

设计和决策基本是我自己主导的——比如整个信任边界怎么划、状态机怎么设计——具体代码是我和 Claude Code 协作写的，我来判断该不该接受它写的东西。

过程中遇到的最具体的一个问题是审计脚本在 Windows 换行符（CRLF）文件上会把所有 frontmatter 字段解析丢失，本质原因是 JavaScript 正则里的 . 不匹配 \r，导致逐行匹配永远走不到字符串末尾。我的修复不是去改正则本身，而是在读文件的时候统一把换行符归一化。这里比较讲究的一点是我没有直接提交一个 CRLF 格式的文件当测试用例，因为 git 会自动把它转回 LF，等于测试永远测不到这个 bug——所以我把测试改成在程序运行时动态拼出 CRLF 内容，还把 Windows 加进了 CI 矩阵，防止这类问题以后又悄悄回归。

这个项目现在的边界也很清楚：它对 Agent 行为的约束都停留在流程和提示词层面，没有代码去强制 Agent 真的遵守，比如'必须打开过文件才能引用'这条规则，本质上是自觉执行，不是代码校验；跨项目、跨其他 Agent 的兼容性目前也只做了文档层面的适配说明，没有真正跑通验证过。这些是我很清楚的下一步方向，不是我没意识到的漏洞。"
