# 📌 泛型基础：为什么需要泛型 + 第一个泛型函数

> Topic: [[TS 类型系统够用版]] · Tier: 1 · 难度: ⭐

## 📖 0. 术语扫盲

| 英文 | 中文 | 一句话解释 |
|---|---|---|
| Generic | 泛型 | 写函数/类型时先不指定具体类型，用 `T` 占位，调用时再决定 |
| `any` | 任意类型 | 关掉类型检查的"逃生舱"，TS 不再帮你查这个值的类型 |
| `unknown` | 未知类型 | 比 any 安全——能接收任何值，但用前**必须先收窄类型** |
| Type parameter | 类型参数 | 那个 `<T>` 里的 `T`，像函数参数，只不过传的是"类型"而非"值" |
| Type inference | 类型推断 | TS 看你传的值，自己猜出 `T` 是什么类型 |

## 📝 1. 实例代码

```typescript
// 代码读法说明：
// 第一段展示 any 的坑；第二段展示写死类型的坑；第三段泛型两者兼得。

// ❌ 方案一：any —— 通用，但丢了类型
function identityAny(arg: any): any {
  return arg;
}
const a = identityAny("hello"); // a 是 any，不是 string
a.toFixed(2); // ✅ 不报错，但 string 没有 toFixed → 运行时崩 💥

// ❌ 方案二：写死类型 —— 类型安全，但不通用
function identityString(arg: string): string {
  return arg;
}
identityString(123); // ❌ number 不能传给 string；处理 number 得再抄一个函数

// ✅ 方案三：泛型 —— 通用 + 类型安全
function identity<T>(arg: T): T {
  return arg;
}
// <T>    = 声明一个类型占位符 T
// arg: T = 参数类型是 T
// ): T   = 返回值类型也是 T（返回一个"值"，不是函数！）

const s = identity("hello"); // TS 推断 T = string，s 是 string ✅
const n = identity(42);      // TS 推断 T = number，n 是 number ✅
// s.toFixed();              // ❌ 立刻标红：string 没有 toFixed
```

⚠️ 易错点：
1. `): T` 是**返回值的类型**，不是"返回一个函数"。返回类型里看到 `=>` 才是返回函数。
2. `<T>` 声明位置别漏：必须在函数名后、参数括号前。漏了直接写 `arg: T` → "找不到名字 T"。
3. 泛型是**编译期**的东西，编译成 JS 后 `<T>` 全部消失，运行时零开销。别以为它把类型信息带到运行时。

## 🎯 2. 使用场景

1. **SaaS 场景**：`first<T>(arr: T[]): T | undefined` 取数组首元素。传 `User[]` 进去，取出来 TS 知道是 `User`，直接 `.name` 有提示——泛型保住了元素类型。
2. **Fintech 场景**：`wrapResponse<T>(data: T): { code: number; data: T }`。无论包装订单、用户还是交易记录，外层结构统一，内层 `data` 类型精确保留，金额字段拼错会报警。
3. **E-commerce 场景**：React 的 `useState<Product[]>([])` 就是泛型，让 `setState` 只接受 `Product[]`，乱传别的类型立刻报错。

## 🔄 3. 可替代技术

| 替代方案 | 优势 | 劣势 | 何时用 |
|---|---|---|---|
| `any` | 写起来最快 | 彻底关掉类型检查，等于没用 TS | 快速原型/过渡期，生产代码尽量别 |
| `unknown` | 比 any 安全，强制你用前检查 | 每次用都要 if 判断/断言，啰嗦 | 接收"真不知道是什么"的外部输入（`JSON.parse`） |
| 函数重载 | 每种类型可精确定义签名 | 类型一多，重载签名爆炸 | 类型组合少且固定、每种行为不同 |
| **泛型 `<T>`** | 通用 + 类型安全，一份代码通吃 | 语法初看有门槛 | **绝大多数"通用容器/工具函数"的默认选择** |

**选型决策树**：
- 如果 真不知道传进来是什么（外部输入） → `unknown`，用前强制检查
- 如果 知道是有限几种、每种行为不同 → 函数重载
- 如果 只是临时糊一下/原型 → `any`（留 TODO 还债）
- 如果 想"通用 + 保留类型"（最常见）→ **泛型 `<T>`**

## 💼 4. 面试题

### 题 1（🟢 基础）
**Q**: `any` 和 `unknown` 有什么区别？
**核心 points**: `any` 关闭检查、会向下污染；`unknown` 接收任意值但用前必须收窄；`unknown` 是 type-safe 的 `any`。
**追问方向**:
1. 为什么说 `any` 会"传染"？（变量沾上 any，后续操作全不检查）
2. `JSON.parse` 返回什么类型？怎么处理？（返回 any，最佳实践是断言成 unknown 或具体类型 + 校验）
**加分回答**: "团队里我会在 ESLint 开 `no-explicit-any`，强制用 `unknown` + Zod 校验外部数据。any 不是类型，是**承诺不再有类型**——它把风险从编译期推迟到线上。"

### 题 2（🟢 基础）
**Q**: 什么是泛型？用一个例子说明它解决了什么问题。
**核心 points**: 泛型 = 类型的参数化；解决"通用性"和"类型安全"的矛盾；类型在调用时才确定。
**追问方向**:
1. 泛型编译后还存在吗？（不存在，类型擦除，运行时零开销）
2. `identity<T>(arg: T): T` 和 `identity(arg: any): any` 区别？（前者保留并**关联**输入输出类型，后者完全丢失）
**加分回答**: "泛型的本质是在类型层面**建立输入和输出的关联**。any 版输入输出类型是断开的，泛型版是绑定的——传 string 进去能一路推导出 string 出来。这种可追踪性是大型代码库可维护的基础。"

## 🧪 5. 小测验

> 补全 `lastElement`，让它对任意类型的数组都类型安全，并说明 `x`、`y` 的类型。

```typescript
function lastElement___(arr: ___): ___ {
  return arr[arr.length - 1];
}
const x = lastElement([1, 2, 3]);
const y = lastElement(["a", "b", "c"]);
```

<details>
<summary>展开答案</summary>

```typescript
function lastElement<T>(arr: T[]): T {
  return arr[arr.length - 1];
}
const x = lastElement([1, 2, 3]);       // x: number
const y = lastElement(["a", "b", "c"]); // y: string
```
更精确的返回类型是 `T | undefined`——空数组时返回 undefined。面试主动提这个 = Senior 思维。
</details>

## 🌟 Senior 思维彩蛋

`any` 在 code review 里是**红线信号**。Mid 看到 any 觉得"能跑就行"，Senior 会问三件事：这个 any 哪来的（外部输入还是偷懒）？能不能换 `unknown` + 校验？它会污染下游多少代码？生产事故里相当一部分"诡异的 undefined"，根子就是某个 any 撕开了类型检查的口子。**any 不是类型，是关掉了类型。**
